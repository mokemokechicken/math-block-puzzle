# 足し算・引き算ブロックパズル コンセプト

## 概要

小学校低学年向けの、足し算と引き算を自然に練習できる Web パズルゲーム。

今回のスコープは、3 ブロックの足し算・引き算と、保証正解の配置方針に置く。掛け算モードや割り算モードは将来候補とし、初期実装には含めない。

数字のブロックが `w x h` の盤面に並び、プレイヤーはマウスまたはスマホのタッチ操作で、縦または横に連続したブロックをドラッグして選ぶ。選んだブロック列が等式として成立すると、ブロックが消え、消えた場所から成立した式が浮き上がる。

ゲームの目的は、計算問題を解かされている感覚ではなく、「数字の並びから成立する式を見つける」感覚で、足し算・引き算への慣れを作ること。

このゲームは、スコアを競うことよりも、反射的に計算できる感覚を育てることを重視する。正解した瞬間の気持ちよさと、浮き上がる式の視覚記憶によって、数字の関係が自然に目に残る体験を目指す。

## 基本ルール

- 盤面には数字ブロックが格子状に並ぶ。
- プレイヤーは縦または横に連続するブロックをドラッグして選択する。
- 選択順がそのまま式の順番になる。
- 最後に選んだブロックを答えとして扱う。
- 最後以外のブロックを左辺として計算し、等式が成立したら成功。
- 成功したブロックは消える。
- 消えたブロックの場所から、成立した式が短く浮き上がる。
- 消えた場所には、上からブロックが落ちるか、新しいブロックが補充される。

正解判定では、レベル 1 から左から右、右から左、上から下、下から上のすべてを許可する。せっかく見つけた正解を、方向だけを理由に不正解にはしない。

## 数字の範囲

ブロックの数字は基本的に `1` から `18` を使う。

`0` は基本的には出さない。`7 + 0 = 7` のような式が増えると、探索の面白さや計算練習の密度が下がりやすいため。

`5 + 7 = 12` のように、答えが 10 を超える式は普通に扱う。低学年向けでも、10 をまたぐ足し算に自然に慣れることを重視する。

## 入門モード

入門モードでは、選べるブロック数を 3 個までにする。

### 足し算

```text
2 -> 5 -> 7
```

この場合は `2 + 5 = 7` として成立する。

### 引き算

```text
9 -> 4 -> 5
```

この場合は `9 - 4 = 5` として成立する。

引き算は順番が重要なので、次の選び方は成立しない。

```text
4 -> 9 -> 5
```

`4 - 9 = 5` ではないため不成立。

## 発展方針

発展では、正解判定の方向を増やすのではなく、保証正解の配置方向を増やす。最初は左から右、上から下のように見つけやすい方向へ保証正解を置き、慣れてきたら右から左、下から上の保証正解も混ぜる。

4 項モードは今回の方針から外す。`a + b + c = d` や `a + b - c = d` は式としては自然だが、盤面探索の負荷が上がり、低学年向けの「反射的に計算できる感覚」から少し離れやすいため。

掛け算モードや割り算モードは将来候補として残すが、初期実装では扱わない。

## 難易度設計

### レベル 1

- 3 ブロックのみ。
- 足し算だけ。
- 正解判定は全方向。
- 保証正解は左から右、上から下に配置する。
- 答えは 18 以下。
- 盤面サイズは小さめ。例: `4 x 4`。
- 盤面内に成立する式を最低 4 個保証する。

### レベル 2

- 3 ブロックのみ。
- 足し算と引き算。
- 正解判定は全方向。
- 保証正解は左から右、上から下を中心に配置する。
- 答えは 18 以下。
- 盤面サイズは `5 x 5` 程度。
- 盤面内に成立する式を最低 4 個保証する。

### レベル 3

- 3 ブロックのみ。
- 足し算と引き算。
- 正解判定は全方向。
- 保証正解に右から左、下から上も混ぜる。
- 答えは 18 以下。
- 盤面内に成立する式を最低 3 個保証する。

### レベル 4

- 3 ブロックのみ。
- 足し算と引き算。
- 正解判定は全方向。
- 保証正解はすべての方向に配置しうる。
- Hard モードとして、保証される正解数を少なめにする。
- 盤面内に成立する式を最低 2 個保証する。

## UX 方針

- ドラッグ中に、選択中の式を画面上に表示する。
- 成功時は、選んだブロックが一瞬つながって光る。
- ブロックが消えた場所から、成立した式がふわっと浮き上がる。
- 式は少し残ってから消える。
- 失敗時は強いペナルティを避ける。
- 低学年向けなので、制限時間やスコア競争よりも「式を見つける楽しさ」を中心にする。
- 計算問題の UI ではなく、パズルとして気持ちよく触れることを重視する。

このゲームの中心体験は、「見つけた」「つながった」「消えた」「式が見えた」の連続にする。

### 正解アニメーション

正解時の基本演出は次の流れにする。

1. 選択したブロックが順番につながって光る。
2. ブロックが小さく弾ける、または吸い込まれるように消える。
3. 消えたブロックの位置から、成立した式が浮き上がる。
4. 式が少し残ってからフェードアウトする。
5. 新しいブロックが落ちてくる。

例:

```text
5 + 7 = 12
```

この式は、正解したあとに盤面上へ短く表示する。プレイヤーがなぞった数字と、成立した計算を視覚的につなげるため。

### ヒント

ヒントは答えを直接教えるのではなく、探索を支える形にする。

- しばらく正解がないと、正解候補に含まれるブロックを 1 つだけ薄く光らせる。
- さらに時間が経つと、答えブロックを薄く光らせる。
- それでも詰まる場合は、正解ライン全体を薄く光らせる。
- 最終ヒントでは、成立する式を表示する。

ヒントは失敗の補償ではなく、考え続けるための支えとして扱う。

## 盤面生成方針

完全ランダムにはしない。難易度ごとに、盤面内に保証される正解数を変える。

- 盤面生成時に、難易度ごとの最低正解数を満たすように成立式を埋め込む。
- 残りのマスは `1` から `18` の数字で埋める。
- プレイヤーが成功するたびに、新しい成立候補を補充する。
- 入門モードでは、常に最低 4 個程度の解ける式が盤面にある状態を目指す。
- Hard モードでは、保証される最低正解数を 3 個または 2 個に下げる。

埋め込んだ式以外に、偶然成立する式が発生することは許容する。プレイヤーにとっては発見の余地になるため。

生成手順の基本形は次の通り。

1. 空の盤面を作る。
2. 難易度に応じた数だけ、成立式を縦または横の連続マスに埋める。
3. 残りのマスを `1` から `18` の数字で埋める。
4. 盤面をスキャンして、成立式の数を数える。
5. 最低正解数を満たしていなければ、再生成または追加で式を埋める。

盤面スキャンでは、正解判定用には全方向を確認する。一方で、難易度ごとの最低保証数を確認するときは、レベルごとに指定された保証配置方向だけを数える。

例えばレベル 1 では、実際の正解判定は右から左、下から上も許可する。ただし「最低 4 個の正解候補を保証する」という条件を満たすための埋め込みは、左から右、上から下の方向に置く。

## 配布方針

Web アプリとして実装し、GitHub Pages で配布する。

初期実装では、静的ファイルだけで動作する構成を優先する。

- `index.html`
- `src/`
- `assets/`
- `docs/`

ビルドツールを使う場合も、最終成果物は GitHub Pages でホストできる静的アセットにする。

## 最初に作るもの

1. `5 x 5` の盤面。
2. 数字ブロックの表示。
3. マウスとタッチでの縦横ドラッグ選択。
4. 3 ブロックの `a + b = c` 判定。
5. 成功時のブロック消去と、消えた場所から浮き上がる式表示。
6. 消えたブロックの補充。
7. 最低 4 個の正解候補を保証する盤面生成。
8. レベル 1 から全方向のドラッグを正解判定する。
9. レベル 1 の保証正解は左から右、上から下に配置する。
10. 追加レベルでは、保証正解の配置に右から左、下から上も混ぜる。
